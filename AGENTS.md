# Инструкции для агента разработки на Go (Golang)

## Контекст проекта и технологии
- Язык программирования: Go (актуальная стабильная версия).
- Система сборки и зависимостей: Go Modules (`go.mod`).
- Стиль архитектуры: Идиоматичный Go, упор на простоту, читаемость и высокую производительность без избыточных абстракций.

## Правила написания кода (The Go Way)

### 1. Форматирование и стандарты
- Весь генерируемый код должен строго соответствовать стандартам `gofmt` и `go vet`.
- Именовать переменные, функции и структуры в стиле camelCase или MixedCaps. Экспортируемые элементы (публичные) всегда начинать с заглавной буквы, неэкспортируемые (приватные) — со строчной.
- Избегать сокращений, ухудшающих читаемость, но использовать стандартные для Go идиомы (например, `r` для reader, `w` для writer, `ctx` для context.Context).

### 2. Обработка ошибок (Error Handling)
- Ошибки — это значения. Всегда проверять возвращаемые ошибки явно через `if err != nil`. Никогда не игнорировать ошибки с помощью `_`.
- Передавать ошибки вверх по стеку, оборачивая их контекстом с помощью `fmt.Errorf("doing something: %w", err)`.
- Использовать `panic` исключительно для критических сбоев инициализации приложения (например, не удалось прочитать обязательный конфиг при старте). В бизнес-логике паники запрещены.

### 3. Конкурентность (Concurrency)
- Запускать горутины (`go func()`) только тогда, когда четко понятен их жизненный цикл и способ их контролируемой остановки.
- Для синхронизации и передачи данных отдавать приоритет каналам (`chan`) и пакету `context` (для отмены таймаутов).
- Если требуется простая синхронизация памяти, использовать пакет `sync` (`sync.Mutex`, `sync.WaitGroup`, `sync.Once`). Избегать состояния гонки (Data Races).

### 4. Архитектура и структуры данных
- Проектировать интерфейсы на стороне потребления (consumer side), а не на стороне реализации. Интерфейсы должны быть маленькими (часто 1-2 метода, как `io.Reader`).
- Избегать глубокой вложенности пакетов и создания пакетов ради пакетов (вроде `utils` или `helpers`). Группировать код по функционалу, а не по типам слоев.
- Первым аргументом в функции, выполняющие I/O или сетевые запросы, всегда передавать `context.Context`.

### 5. Тестирование
- Каждая новая фича или пакет должны сопровождаться модульными тестами в файлах `*_test.go`.
- Использовать стандартный пакет `testing`. Предпочитать табличные тесты (Table-driven tests) для проверки различных сценариев работы одной функции.

## Алгоритм работы над задачами

### В режиме Plan Mode (`Tab`)
1. Проанализировать структуру существующих пакетов и файлов `go.mod` / `go.sum`.
2. Расписать пошаговый план: какие типы (struct), интерфейсы и функции будут добавлены.
3. Описать, как будет устроена обработка ошибок и логирование в рамках новой фичи.

### В режиме Build Mode (`Tab`)
1. Генерировать чистый Go-код, готовый к компиляции.
2. Тщательно комментировать экспортируемые функции и типы (документирующие комментарии в начале строки).
3. После написания кода запускать проверку синтаксиса и тесты с помощью команд вроде `!go test ./...` для верификации правок.

## Правила работы с Git
- **Никогда не делать commit автоматически**, если пользователь не попросил об этом явно.
- Если внесены логические или завершенные изменения, **предложить** пользователю сделать коммит, но не выполнять его самостоятельно без подтверждения.

## Компиляция
- Всегда компилировать проект под текущую ОС (Windows), чтобы пользователь мог сразу проверить изменения после доработок (создавать `mycalendar.exe` в корне проекта).
